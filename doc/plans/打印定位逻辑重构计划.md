# 打印定位逻辑重构计划

> **目标**：将 `console.log` 插入位置的计算，从「16 种表达式场景枚举」重构为「语句锚定模型」。
> **状态**：已完成 ✅（阶段 0–3e 全部交付，旧引擎已下线）
> **关联模块**：`src/print/locator.ts`（定位器）、`src/print/ast.ts`（解析基元）、`src/print/handler.ts`（编辑渲染）
> **前置阅读**：[打印需求计划.md](./打印需求计划.md)

---

## 一、重构动机

### 1.1 现状

当前 [`findInsertionLine`](../../src/print/ast.ts) 按优先级串行匹配 16 个 `check*` 函数确定插入行号。该方案存在四个结构性缺陷：

| # | 缺陷 | 根因 |
|---|------|------|
| 1 | 场景永远补不完，维护成本线性增长 | 建模在「表达式形状」这一**无限、开放**的维度上 |
| 2 | 同行同名变量无法区分，多行语句易误命中 | 接口只传 `selectionLine`（行号），**丢失列/offset 信息** |
| 3 | 缩进漂移 | AST 层只返回 `line`，缩进由 handler 启发式猜测 |
| 4 | 性能冗余 O(场景数 × 文件大小 × 选区数) | 每个 check 独立全量 `walkAST`，父子表重复构建，多光标重复 `parseCode` |

### 1.2 关键洞察

- **表达式形状无限，但插入落点高度收敛**：场景 5/6/7/8/9/10/13 对格式良好的单语句计算的是同一落点，是同一条规则的重复实现。
- **语句容器是有限封闭集**：`Program.body`、`BlockStatement.body`、`SwitchCase.consequent`、`StaticBlock.body`。任何合法插入点，都是「某语句容器的语句序列中的某下标」。
- **成员容器不是插入目标**：`ClassBody.body` 装的是 `PropertyDefinition/MethodDefinition` 而非语句，**类体直属层无法合法放置 `console.log` 语句**。因此选中类字段（无论 public `config` 还是 private `#private`）统一按**不支持（SKIP）**处理（已拍板，见 2.8）；现有代码「字段后直插」会产出非法 JS，属于需修正的隐性 bug。
- 参考 turbo-console-log 的「语句锚定 + 产生位/使用位二分 + 函数上下文特判」理念，但用 AST 替代其「行文本 + 括号计数」，边界更严谨。

## 二、目标架构：语句锚定模型（Statement-Anchored Model）

> 核心范式：不再问「这是什么表达式」，而是问「被选中的节点挂在哪条语句上，这条语句属于哪个语句容器」。

### 2.1 定位算法（三步，替代 16 个 check）

```
第一步：精确命中被选中节点 target
  用 [selectionStartOffset, selectionEndOffset]（offset，而非行号）
  在 AST 中找到「被选区完整包含 / 最贴合选区」的最小节点。
  → 消除同行同名歧义与多行语句误命中。

第二步：沿祖先链上爬，找到锚语句 stmt 与语句容器 container
  单次构建 parentMap，从 target 向上，
  直到 parent 为语句容器（其某字段是语句数组）为止；
  stmt = 容器中包含 target 的那条直接子语句。

第三步：由 (target, stmt, container) 决定 placement
  查决策表输出三类插入位之一（见 2.2）。
```

### 2.2 placement 决策表（一条默认 + 三个例外 + 一个第三态）

| Placement | 触发条件（target 在锚语句中的角色）| 语义 | 覆盖旧场景 |
|-----------|-----------------------------------|------|-----------|
| **AFTER_STMT**（默认）| 产生位：`VariableDeclarator.id`、`AssignmentExpression.left`、被赋值的表达式 | 变量在此获得值，插到语句后才有值可打 | 1/3/4/5/6/7/8/9/10/12/13 |
| **BLOCK_HEAD**（例外一）| 函数参数（含解构 / `TSParameterProperty`）| 参数在函数体内才可见，插到体首行 | 2 |
| **BLOCK_HEAD**（例外二）| 循环头：`for` / `for-in` / `for-of` 的 init/left 声明 | 声明在循环体内才有值，插到循环体首行 | 13（for-init 分支）|
| **BEFORE_STMT**（例外三）| 使用位：条件表达式、`return` 实参、游离读取 | 打印已存在的值，插到语句前符合调试直觉 | 11/14/15/16 |
| **SKIP**（第三态）| 类型层：`TSEnumMember.key`、`TSPropertySignature.key`、接口属性等编译期声明 | 无运行时值，不可打印 | 现由游离场景显式跳过 |

> 「1 默认 + 2 例外」的初始设想经与现有已修复场景对照后修正为「**1 默认 + 3 例外 + 1 第三态**」——循环头是独立于函数参数的第二种块首插入，类型层则完全落在产生位/使用位二分之外。

判定「产生位 vs 使用位」只需检查 target 在**直接父节点**中的角色，局部、正交、可独立单测，不依赖规则先后顺序。

### 2.3 normalize-then-insert（补块归一化）

无块体的形态（`() => expr`、无花括号的 `if(x) stmt`、`for(...) stmt`）统一走预归一化，替代散落各处的特判：

```
若目标落点需要 BlockStatement 而当前是表达式体 / 单语句：
  先生成「补花括号」编辑：expr body → { return expr }，single stmt → { stmt }
  再在归一化后的块内计算插入点。
```

一个 `ensureBlock` 步骤替代：箭头函数表达式体转换、空函数体判断、for-init 补块。

> **坐标一致性（洞 4）**：`normalize` 补 `{ }` 会改变文本长度，使其后所有 offset 偏移。因此 `normalize` 编辑与 log 插入编辑**必须均以原始 code 坐标表达**，交由 VSCode 在同一 `WorkspaceEdit`/`editBuilder` 中原子合并；严禁先 apply normalize 再用旧 offset 算插入点。

> **已拍板：改写用户源码结构可接受**。将 `() => expr` 补为 `() => { return expr }` 会改动用户函数写法，副作用大于「仅插一行 console.log」；经确认这是合理且必要的改动（与 turbo-console-log 一致，且为现有散落特判的统一归它），予以保留。**多选批量下含 normalize 的完整防漂移对策见 2.9（洞 5）。**

### 2.4 结构化返回值（根除缩进漂移）

AST 层返回结构化锚点，缩进对齐锚点一并交给 handler，删除 handler 的相邻行缩进启发式：

```ts
interface InsertionResult {
  /** 基于 AST offset 的精确插入位置 */
  offset: number
  /** 缩进对齐锚点：指向一个现成行的 offset，handler 复制该行的缩进前缀 */
  indentRef: number
  /** 插入分类 */
  placement: 'AFTER_STMT' | 'BLOCK_HEAD' | 'BEFORE_STMT'
  /** BLOCK_HEAD 时是否需在 indentRef 行缩进基础上再加一个缩进单位 */
  indentOneLevelDeeper?: boolean
  /** 需补花括号时的前置编辑 */
  normalize?: BlockEdit
}
```

> **命名冲突须处理**：现有 `findInsertionLine` 的返回类型已占用 `InsertionResult` 之名（旧结构含 `line`，见 `ast.ts`）。本结构为**替换**而非新增：阶段 3 下线 16 check 时旧 `InsertionResult` 一并移除；阶段 2 新旧引擎并行期，新结构需临时改名（如 `InsertionAnchor`）避免类型冲突。

> **indentRef 语义按 placement 分流（洞 1）**：声称「根除漂移」不能只给一个 offset 就了事。
> - **AFTER_STMT / BEFORE_STMT**：`indentRef` 指向同级锚语句行，handler 直接复制该行缩进。
> - **BLOCK_HEAD**：目标缩进 = 块头行缩进 + 一个缩进单位。若块内已有语句，`indentRef` 指首条内层语句行（现成正确缩进）；若为**空块或 normalize 新建块**（`function g(){}`、`() => expr`）则无现成内层行可抵，置 `indentOneLevelDeeper=true`，由 handler 以块头行缩进加上 editor 配置（`tabSize`/`insertSpaces`）的一个缩进单位得出——这是 editor 确定量，不属于被禁止的「猜相邻行」。

### 2.5 selection → node 解析（承重墙，需单独设计）

> 现有功能对**任意选中文本**工作（`selectedText` 字符串），用户可选 `user.first`、`a + b`、半个 token。新模型第一步就依赖能拿到确定的 target 节点，因此解析规则是整个模型的承重墙，必须显式定义、不能以「最贴合节点」四字盖过。

解析规则（优先级由高到低）：

1. **区间完全包含**：取 `[start, end]` 被选区完整包含、且自身区间最小的节点。
2. **成员路径整体优先**：若选区文本含 `.` 且与某 `MemberExpression` 的源文本逐字相等，命中该成员表达式整体（**迁移现有确定修复**：避免选 `token` 误命中 `this.tokenValue`，见 `createIdentifierSearcher`）。
3. **解包下钻**：命中节点若为 `TSAsExpression/TSTypeAssertion/TSNonNullExpression/ParenthesizedExpression/ChainExpression` 等包裹（与现有 `UNWRAP_TYPES` 集合一致，共 5 种），递归解包到内层实表达式（复用 `unwrapTypeAssertion`）。
4. **角色上推**：得到 target 后，向上找到它在锚语句中扮演的角色（产生位/使用位/参数/循环头/类型层），喂入 2.2 决策表。

该解析器需作为**独立可单测模块**实现，不得与 placement 决策耦合。

> **定位与打印内容分离（洞 3）**：`findInsertion` **只管定位**（返回 offset/indentRef/placement），**不返回打印文本**。打印内容始终是用户**原始选中文本**（`document.getText(selection)`），由 handler 用 `parseConsoleLogTemplate` 生成。本节 refine（选 `token` → 命中 `this.token`）、解包（`as`/`!`）**仅影响落点，不得改变打印的文本**，否则会引入未声明的行为变更。

### 2.6 offset → 插入点归一化

> offset 精确，但「节点 offset」不等于「可直接插入的 offset」：行/列的问题不是消失了，而是被搬到了这里。必须定义归一化规则，否则会插出行内断裂的坏代码：

- **AFTER_STMT 跨到行尾换行之后**：`stmt.end` 可能落在行中（`const a=1; const b=2` 选 `a`）或行尾注释之前（`const x = 1 // 注释`）；插入点需从 `stmt.end` 向后扫到当前行的行尾换行符之后。
- **一行多语句**：需确认插到整行之后而非分号中间，与旧代码「整行 +1」的粗粒度行为对齐。
- **BEFORE_STMT / BLOCK_HEAD 取行首 offset**：插入点对齐到锚行/块首行的行首，缩进由 `indentRef` 给定。

### 2.7 已知退化缺口（必须主动决策，不能默认继承）

以下是现有跑绿但新模型若按现状会挂的点，已分别由 2.1、2.2、2.5、2.6 补齐：

| 缺口 | 风险 | 处置 |
|------|------|------|
| 类字段 `protected config = new Foo()` / 私有字段 `#private = 1` | 类体直属层放语句是**非法 JS**（现有隐性 bug）| **已定：方案 A 不支持，锚容器为 ClassBody 时 SKIP**（见 2.8）|
| for-init `for(let i=0;...)` 选 `i` | 默认 AFTER_STMT 会插到整个 for 之后 | 2.2 例外二：循环头 → 体首行 |
| 选中成员路径 `this.token` | offset 命中层级歧义 | 2.5 规则 2：成员路径整体精确匹配 |
| 一行多语句 / 行尾注释 | offset 插入点落在行中 | 2.6 跨到行尾换行之后 |

### 2.8 已决策：类字段 / 私有字段的 log 处置 → 采用 A（不支持）

> 根因：`ClassBody` 只能容纳 `PropertyDefinition/MethodDefinition`，**不能直接放语句**。选中类字段（`config` / `#private`）后，无论插在字段前还是字段后，只要落在类体直属层就是非法 JS。现有代码照插会产出坏代码，必须二选一。

| 方案 | 做法 | 优点 | 代价 |
|------|------|------|------|
| **A：不支持（✅ 已采用）** | target 的锚容器是 ClassBody 时，**静默不插入**（遵循现有解析失败静默降级策略，无用户提示）| 绝不产出坏代码；实现最简单；与裁剪边界一致 | 用户选字段按快捷键无反应，可能困惑 |
| B：插进构造器首行（未采用）| 若类存在 `constructor`，将 log 插入构造器体首行 | 产出合法且能运行的代码 | 字段初始值与构造器执行时机不同，语义可能误导；无构造器时需退回 A |

**✅ 决定：采用 A（不支持）**。理由：与项目「静默降级、不过度设计、词法可预测即达标」的健壮性边界一致；方案 B 引入的「字段声明位 vs 构造器执行位」语义错位反而违背调试直觉（给错值比不给值更危险）。若后续有强需求再补 B，作为 A 之上的渐进增强。

**实现要点**：`decidePlacement` 中，当 target 向上爬时遇到的容器是 ClassBody（而非语句容器），直接返回 SKIP；`findInsertion` 返回 null，handler 不做任何编辑。

### 2.9 多选批量打印不变式（洞 5、横切约束）

> 现有功能支持多光标/多选区一次性批量打印。每个选区的定位结果各自独立地基于**原始 AST/代码**算出，漂移只在「批量应用编辑」时发生。**必须把「多选是否漂移」当作每条 placement / normalize 规则的验收项，而非个案。**

**安全基线**：若一批全是**互不重叠的纯插入**（无 normalize），VSCode 对原始文档快照按 offset 应用，顺序无关、绝不漂移。

**三类失效模式与对策**：

| 失效模式 | 触发 | 对策 |
|---------|------|------|
| ① 结构扩张使兄弟锚点失效 | 某选区触发 normalize 补块，撑大了另一选区所在语句/节点（如同一箭头函数选 `fn` + `expr`）| **两阶段（方案甲）**：先 apply 全部 normalize → 重新 parse 一次 → 在归一化后的 AST 上重算所有 log 插入 offset |
| ② 同 offset 多插入 | 多个选区解析到同一锚点（如 `function g(a,b){}` 同选 `a`、`b` → 同为体首行；`b+c` 同选 `b`、`c` → 同为语句前）| 保证每个都是**独立非重叠 insert**（非 replace）；同 offset 按**稳定顺序**（原选区顺序）依次插入，不得合并 |
| ③ 编辑区间重叠 | 一个选区的编辑范围与另一个重叠（如成员路径整体 + 其内层标识符同选）| 检测重叠 → 串行化处理或对后者静默降级，绝不产出重叠编辑（VSCode 会抛错/未定义）|

**落地纪律**：`handler` 批量应用统一按 **offset 降序**（洞 2）；批内**只要含任一 normalize，整批走两阶段**；应用前对全部编辑做**重叠自检**。

**验收要求**：每新增一种 placement 或 normalize 形态，必须补一条「与另一选区同批」的用例（同锚点、被撑大、区间重叠三种至少覆盖相关者）。

## 三、接口契约变更

### 3.1 handler → ast（传入 offset）

```ts
// 旧
findInsertionLine(code: string, variableName: string, selectionLine: number, fileExtension?: string): { line: number, ... }

// 新
findInsertion(
  code: string,
  selection: { startOffset: number; endOffset: number },
  fileExtension?: string
): InsertionResult | null
```

- `handler.ts` 由 `selection.active.line` 改为 `document.offsetAt(selection.start / selection.end)`。
- 模板文件（Vue/HTML/Svelte/Astro）script 提取时，offset 同步做 `byteOffset` 换算，复用现有 `adjustASTLocations` 机制。
- **模板脚本块定位不能只靠 offset**：现有 `parseCode` 借 `selectionLine` 经 `extractVueScript/extractSvelteScript` 选定选区所在的 `<script>` 块（Vue 可有 `<script>` + `<script setup>` 多块）。切 offset 后须保留块定位能力——handler 内把 offset 转回 line 传入提取，或改造提取函数接受 offset；否则模板文件会选错 script 块。

### 3.2 handler 缩进逻辑

- 删除 [`handler.ts`](../../src/print/handler.ts) 中 `atIndent / aboveIndent` 比较段。
- 改为读取 `indentRef` 处行的缩进前缀（复用 `getLineIndent`）。

## 五、正确性基准用例清单

> 以 VSCode 真实插入效果为准。（本文件原 `[验证]` 项已在评审中逐条消解，现存用例均为定稿期望。）

### 5.0 现有测试基准核对结论（阶段 0）

逐套件核对现有 123 条 print 测试，结论：**未发现锁定错误行为的用例**，可作为可信回归基准。

- **换行一致性套件**（`ast-linebreak`，6 条）：锁定的是 U+2028/U+2029/CRLF/单 `\r` 与 VSCode 行模型对齐的**修复**，非 bug。
- **选区健壮性套件**（`ast-selection`，5 条）：锁定的是首尾空白 trim、成员路径不误匹配的**修复**，非 bug。
- **16 场景套件**（`ast.test` / `ast-extended`）：用例选中的均为**产生位变量**（`myFn`/`result`/`value`/`config`/`x`/`sum`），期望均为 AFTER_STMT；return 上下文选中使用位（如 `obj`）期望插入 return 前——与新语句锚定模型的 AFTER_STMT / BEFORE_STMT 判定一致。

**阶段 1-2 需重点关注的预期变更点（非回归，属新能力/新行为，阶段 2 diff 回归应将其列入白名单）：**
1. 表达式体箭头函数（`() => expr`）的 normalize 补块输出格式（现有用例仅覆盖块体箭头函数）。
2. 类字段 ClassBody → SKIP（现无测试覆盖，阶段 1 需补用例）。
3. 同行同名 offset 消歧（旧 `line+name` API 无法表达，需基于 `SelectionRange` 新建用例）。

## 六、健壮性边界（避免过度设计）

结合功能裁剪清单，重构严守既定边界，不扩张：

- 维持已裁剪项：不引入 `consoleLogScope`、`consoleLogLevel`、右键菜单、批量删除确认框。
- 维持 console 方法范围限制、console.log 跨行禁用等既有边界。
- **词法可预测即达标**：异步/闭包/回调等运行时动态语义不属于静态插入点范畴；目标是落点词法可预测、不乱跳，而非运行时正确性。
- 作用域判断（如 return 内回调中的选中节点）由「祖先链遇到函数节点即换锚容器」天然表达，无需特判。
- 解析失败沿用现有静默降级策略（`parseCode` 返回 null → 回退 `selectionLine + 1`），不弹用户提示。
- **非空选区为前提**：现有功能对空选区（纯光标）直接跳过（`if (!selectedText) continue`），`findInsertion` 不需处理零宽选区，不属于本次重构扩展边界。
- **JSX/TSX 在解析范围内**：`.jsx/.tsx/.astro` 已启用 jsx 解析（见 `parseCode`），JSX 内标识符按使用位/产生位同规则处理，需纳入用例。

## 七、新旧方案对比

| 维度 | 现状（16 场景）| 语句锚定模型 |
|------|---------------|-------------|
| 建模维度 | 表达式形状（无限）| 语句容器（有限）|
| 新语法覆盖 | 需新增 check | 零代码自动覆盖 |
| 定位精度 | 行号 + 名字，同行同名歧义 | offset 节点级精确命中 |
| 规则耦合 | 16 条有序 | 1 默认 + 3 例外 + 1 第三态，正交可测 |
| 缩进 | handler 启发式猜测 | AST 返回 indentRef，机械对齐 |
| 补块 | 多处特判 | 统一 normalize |
| 性能（单光标）| ~16 次 walk + 5 次父子表 | 1 次 walk + 1 次父子表 |
| 多光标 | 每选区重新 parse | 单次 parse，批量定位 |
